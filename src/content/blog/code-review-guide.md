## 🙋 新人第一次 Code Review 的真实场景

刚入职那会儿，我特别怕 Code Review。每次提交 PR（Pull Request）后，都会紧张地盯着 Slack 频道，等待大佬们的点评。

有一次，我信心满满地提交了一个"完美"的代码：

```javascript
// 我的第一次Code Review提交
function calculateSum(arr) {
  let sum = 0
  for (let i = 0; i < arr.length; i++) {
    sum += arr[i]
  }
  return sum
}
```

评审结果呢？

> 高级工程师老王："能用 reduce 吗？"
> 我："什么是 reduce？"
> 老王："......行吧，先这样吧。"

从那以后，我开始认真研究 Code Review，逐渐从一个"被动挨打"的新人，变成了能够给别人提出有价值建议的老油条。

---

## 🎯 Code Review 的真正目的

首先，我们要搞清楚 Code Review 的目的：

### ❌ Code Review 不是找茬

```
目的 = 找出所有问题 ❌
```

很多人把 Code Review 当成"挑刺大会"，评审者拼命找问题，提交者拼命辩解。这种对抗心态只会让团队氛围越来越紧张。

### ✅ Code Review 是知识共享

```
目的 = 知识共享 + 代码质量提升 + 团队成长 ✅
```

一个好的 Code Review 应该是：

- **评审者**：分享经验，帮助新人成长
- **提交者**：接受反馈，学习新知识
- **双方**：共同提升代码质量

## 📝 10 个 Code Review 避坑技巧

### 技巧 1：看需求，不只看代码

**❌ 错误示范**

```javascript
// 评审者：这个函数为什么要用递归？
// 提交者：因为这样写比较简洁...
// 评审者：递归性能不好，换成循环！
```

**✅ 正确做法**

```javascript
// 评审者：这里用递归实现，有什么特殊原因吗？
// 提交者：这个树结构深度不确定，递归更直观。
// 评审者：了解。不过要注意，当树层级超过1000时可能会栈溢出，需要加个深度限制。
```

**要点**：先理解需求，再讨论实现。不要在不清楚背景的情况下强行要求别人改代码。

---

### 技巧 2：关注逻辑，不只是格式

**❌ 错误示范**

```javascript
// 评审者：你这个缩进不对，4个空格要用2个空格！
// 提交者：...（内心：就这？）
```

**✅ 正确做法**

```javascript
// 评审者：这段业务逻辑我看不太懂，能解释一下为什么这样处理吗？
// 特别是这个判断条件：user.role !== 'admin' && user.status === 'active'
// 是需要同时满足两个条件吗？
```

**要点**：格式问题交给 ESLint/Prettier，Code Review 的时间应该花在刀刃上——业务逻辑和架构设计。

---

### 技巧 3：提出建议，不是下命令

**❌ 错误示范**

```javascript
// 这个写法太烂了，换成XXX
```

**✅ 正确做法**

```javascript
// 这里可以考虑用XXX实现，我觉得会更清晰一些。
// 当然，如果这样做有特殊原因，保持现有实现也没问题。
```

**示例对比**：

```javascript
// ❌ 命令式
// 必须用TypeScript enum，不能用const对象！

// ✅ 建议式
// 这里用const对象定义状态列表，TypeScript支持得很好。
// 如果后续需要更复杂的枚举行为，可以考虑升级为enum。
// 当然目前的实现也完全OK，仅供参考。
```

---

### 技巧 4：review 代码前先 review 自己

每次 review 别人的代码前，先问自己三个问题：

```markdown
1. 我的建议是"必须有"还是"建议有"？
2. 这个建议能解决实际问题，还是只是我的个人偏好？
3. 如果我遇到同样的场景，我会做得更好吗？
```

**好的评审者**：

```javascript
// 这段代码逻辑没问题，但如果未来需要支持XXX，
// 可以考虑预留一些扩展点。
// 仅供参考，不强制修改。
```

**一般的评审者**：

```javascript
// 这样做不对，应该按我的方式来。
```

---

### 技巧 5：学会给代码点赞

Code Review 不只是找问题，也要发现闪光点！

```javascript
// 👍 代码中的亮点要及时肯定
// 哇，这里用策略模式处理不同支付方式，很优雅！
// 👍 注释写得很清楚，一看就懂。
// 👍 这个边界处理很到位，想得很周全。
```

**为什么重要**：

- 新人会更有动力保持好习惯
- 团队氛围会更积极向上
- 发现好代码可以推广到其他地方

---

### 技巧 6：解释"为什么"，不只是"是什么"

**❌ 错误示范**

```javascript
// 这里要用防抖
```

**✅ 正确做法**

```javascript
// 搜索场景建议加防抖，原因：
// 1. 避免用户每次输入都发请求，减少服务器压力
// 2. 减少不必要的接口调用，提升性能
// 3. 典型案例：百度搜索框的延迟搜索效果
//
// lodash的debounce很好用，或者用useDebouncedCallback也可以。
```

---

### 技巧 7：分清主次，不要眉毛胡子一把抓

```javascript
// 评审时给问题分类

// 🔴 P0 - 必须修复（会影响功能）
// 示例：内存泄漏、安全漏洞、严重bug

// 🟡 P1 - 建议修复（影响可维护性）
// 示例：代码可读性差、缺少必要的注释

// 🟢 P2 - 可以优化（锦上添花）
// 示例：变量命名、代码风格、个人偏好
```

**格式建议**：

```markdown
## P0（必须修复）

- [ ] 第 23 行：数组边界没有检查，可能导致越界访问

## P1（建议修复）

- [ ] 第 45 行：这段逻辑可以抽取为独立函数，便于后续维护

## P2（可选优化）

- [ ] 第 67 行：可以考虑用解构赋值，代码更简洁
```

---

### 技巧 8：控制 Review 的范围

**一次 PR 不要太大**：

```markdown
# 建议的 PR 大小

✅ 100-300 行：最佳，太小显得碎片化
⚠️ 300-500 行：可以接受
❌ 500+行：建议拆分

# 如果 PR 太大，可以这样评审

1. 先看整体架构设计
2. 再看关键业务逻辑
3. 最后看边边角角
```

**评审者的心态**：

> 面对一个 500 行的 PR，再有耐心的评审者也会疲惫。所以提交者要控制 PR 大小，这是对评审者的尊重。

---

### 技巧 9：用代码说话

**❌ 错误示范**

```javascript
// 评审者：这里改成XXX更好
// 提交者：怎么改啊？
// 评审者：你应该知道啊
// 提交者：......（内心：你倒是说清楚啊）
```

**✅ 正确做法**

```javascript
// 评审者：这里可以考虑这样改：
//
// // 改前
// if (status === 'success') {
//   doSuccess();
// }
//
// // 改后
// const handlers = {
//   success: doSuccess,
//   error: doError,
//   loading: doLoading,
// };
// handlers[status]?.();
//
// 这样后续新增状态类型更方便。
// 当然如果场景简单，保持现在的写法也没问题。
```

### 技巧 10：评审后的跟进

Code Review 不是评审完就结束了：

```markdown
## 评审后跟进清单

### 作为提交者

- [ ] 认真对待每一条反馈
- [ ] 有疑问及时沟通，不要猜测
- [ ] 修改后记得@评审者确认
- [ ] 感谢评审者的建议

### 作为评审者

- [ ] 及时评审，不要让 PR 等太久
- [ ] 追踪问题是否已修复
- [ ] 对新人的进步给予肯定
```

## 🚀 新人参与 Code Review 的正确姿势

### 1. 提交代码前先自查

```markdown
## 提交前的 Checklist

- [ ] 代码逻辑是否正确？
- [ ] 边界情况是否处理？
- [ ] 变量命名是否清晰？
- [ ] 是否有必要的注释？
- [ ] ESLint/Prettier 是否通过？
- [ ] 单元测试是否覆盖？
- [ ] PR 描述是否清晰？
```

### 2. 把每次评审当成学习机会

```javascript
// 遇到不懂的概念，记录下来，评审后去学习
// 例如评审者提到了"策略模式"，你不懂：
// 1. 先记下来：策略模式 - 需要学习
// 2. 评审后查阅资料
// 3. 弄懂后可以在评论区回复："谢谢，学到了！"
```

### 3. 主动参与评审

```markdown
# 即使是被评审，也要主动看别人的代码

好处：

1. 了解团队的整体代码风格
2. 学习别人的优秀写法
3. 发现自己可能也犯过的错误
4. 建立技术影响力
```

## 💬 总结

Code Review 是团队协作中非常重要的一环，它的目的不是挑刺，而是：

> **共同提升代码质量，分享技术经验，让团队一起成长。**

作为新人：

1. 保持开放心态，接受批评
2. 主动学习，不懂就问
3. 关注逻辑，不纠结格式
4. 学会感谢，肯定别人

作为评审者：

1. 提出建议，不是命令
2. 解释原因，分享经验
3. 分清主次，聚焦重点
4. 多点赞美，营造氛围
